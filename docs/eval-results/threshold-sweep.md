# Threshold sweep — full P/R curve

Generated by `scripts/10_threshold_sweep.py --reranker` over the 50-question v2 golden dataset. Two sweeps: a lower band (0.18–0.30) to find the best F1, and a higher band (0.30–0.50) to find where unanswerable handling kicks in.

Run date: 2026-05-18. Corpus: 686 CDC documents → 2 195 chunks (`all-MiniLM-L6-v2`, 384-dim, cross-encoder reranker on top of top-10 retrieval).

---

## The tradeoff, plainly

|             | threshold = 0.18 | threshold = 0.50 | delta |
|-------------|------------------|------------------|-------|
| Precision@5 | 0.248            | 0.184            | -26%  |
| Recall@5    | 0.380            | 0.325            | -14%  |
| Unanswerable handling | 0/4 (0%) | 4/4 (100%) | +100pp |

**Raising the threshold from 0.18 → 0.50 trades 26% of Precision@5 for going from 0% to 100% on unanswerable handling.**

Whether that trade is worth it depends on the production cost of each failure mode:

- **In a system that's gated by guardrails** (which HealthRAG is — the generator refuses to answer when retrieval is empty), unanswerable handling has high marginal value. Threshold 0.45–0.50 is defensible.
- **In a system where retrieval feeds an LLM that then "graciously declines"** based on its own assessment of relevance, retrieval handling matters less and we can leave threshold low.

The current production default is `0.20`, which optimizes for the first scenario — the LLM's behavior on empty retrieval is the safety net.

---

## What I'd try next

1. **A per-category threshold** — keep 0.20 for answerable categories (single_doc, multi_doc, personalized) and apply 0.50 only when the question intent classifier predicts `unanswerable` / `out_of_scope`. Best of both. Needs a small intent classifier.
2. **Two-stage retrieval** — run threshold 0.20 first; if the top result's similarity is below some "confidence floor" (say 0.30), treat it as unanswerable regardless. This is what threshold-tuning is approximating but in a more principled way.
3. **Confidence calibration** — the cosine similarity from MiniLM is not a probability. A small Platt-scaling layer trained on the golden dataset would convert similarities to "is this chunk *actually* relevant?" probabilities.

## Reproducing this

```bash
# low band
python scripts/10_threshold_sweep.py --reranker --thresholds "0.18,0.20,0.22,0.25,0.28,0.30"

# high band
python scripts/10_threshold_sweep.py --reranker --thresholds "0.30,0.35,0.40,0.45,0.50"
```

Each sweep over 50 questions × 6 thresholds takes ~35 seconds. Zero LLM calls.
